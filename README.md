## Лабораторна робота №1: Основи розгортання веб-застосунків

### Розрахунок варіанту та конфігурація
Номер у списку (N) = 9.

| Параметр | Розрахунок | Значення за варіантом |
| :--- | :--- | :--- |
| **V2** (Застосунок) | `(9 % 2) + 1 = 2` | **Notes Service** — простий сервіс для зберігання текстових нотаток. |
| **V3** (Конфігурація) | `(9 % 3) + 1 = 1` | **Конфігураційний файл** `/etc/mywebapp/config.json` |
| **V5** (Порт та БД) | `(9 % 5) + 1 = 5` | Порт: **5000** <br> База даних: **PostgreSQL** |

### Опис застосунку та локальний запуск
Застосунок написаний на Python (Flask + Gunicorn) і підключається до PostgreSQL. При запуску автоматично виконується скрипт міграції (`migration_script_db.py`) для створення таблиць.

**Як налаштувати середовище та запустити застосунок локально:**
1. Переконайтеся, що встановлено Python 3 та PostgreSQL.
2. Створіть базу даних та користувача в PostgreSQL.
3. Створіть віртуальне середовище: `python3 -m venv venv`
4. Активуйте його: `source venv/bin/activate`
5. Встановіть залежності: `pip install -r requirements.txt`
6. Запустіть міграції: `python migration_script_db.py`
7. Запустіть застосунок: `python app.py`

### Документація API

API віддає дані залежно від того що клієнт очікує (повертає `application/json` або `text/html` залежно від заголовка `Accept` у клієнта).

**Ендпоінти бізнес-логіки:**
* `GET /` — Кореневий ендпоінт. Завжди віддає `text/html` зі списком усіх доступних ендпоінтів бізнес-логіки.
* `GET /notes` — Виводить список усіх нотаток (повертає id та title).
* `POST /notes` — Створює нову нотатку (приймає параметри title та content).
* `GET /notes/<id>` — Виводить повний вміст нотатки (id, title, created_at, content).

**Ендпоінти перевірки стану (Health checks):**
*(Доступні лише локально через порт 5000, заблоковані ззовні через Nginx).*
* `GET /health/alive` — Завжди повертає HTTP 200 з текстом `OK`.
* `GET /health/ready` — Повертає HTTP 200 `OK`, якщо є успішне підключення до БД, інакше HTTP 500 з описом помилки.

### Доступ по SSH
Після розгортання доступні такі користувачі (`ssh <user>@<ip>`):

| Користувач | Пароль | Права та призначення |
| :--- | :--- | :--- |
| `ubuntu` | - | Дефолтний користувач (заблокований після виконання скрипта). |
| `student` | `12345678` | Права `sudo`. Пароль треба змінити при першому вході. Власник файлу `gradebook`. |
| `teacher` | `12345678` | Права `sudo`. Пароль треба змінити при першому вході. |
| `operator` | `12345678` | Обмежений `sudo` (лише керування `mywebapp.service` та reload Nginx). Без доступу до `gradebook` та керування сокетом `mywebapp.socket`. |
| `app` | - | Системний користувач без права входу (shell: `/bin/false`). Від нього працює Systemd-сервіс. |

### Інструкція з тестування системи

Цей розділ описує покроковий процес перевірки працездатності розгорнутої інфраструктури, ендпоінтів та налаштованої безпеки.  В прикладах використовується IP-адреса `192.168.31.247` (Nginx) та локальна адреса `127.0.0.1:5000` (пряме звернення до застосунку).

#### 1. Перевірка Systemd Socket Activation
Перевірка того, що застосунок запускається лише за вимогою.

* **Перегляд статусу сокета:**
  `sudo systemctl status mywebapp.socket`
  *(Очікуваний стан: `active (listening)`).*
* **Перегляд статусу застосунку до першого запиту:**
  `sudo systemctl status mywebapp.service`
  *(Очікуваний стан: `inactive (dead)` — процес вимкнений для економії ресурсів).*
* **Пробудження застосунку:**
  `curl http://127.0.0.1:5000/notes`
* **Перевірка статусу після виклику:**
  `sudo systemctl status mywebapp.service`
  *(Очікуваний стан: `active (running)`).*

#### 2. Перевірка API та Content Negotiation
Перевірка створення нотаток та віддачі даних у різних форматах.

 * **Вивід списку всіх нотаток:**
`curl -X GET http://192.168.31.247/notes/ -H "Accept: application/json"`

* **Створення нової нотатки (POST):**
   ```bash
  curl -X POST http://192.168.31.247/notes \
       -H "Content-Type: application/json" \
       -H "Accept: application/json" \
       -d '{"title": "My first test note", "content": "Everything is working perfectly!"}'
   ```
* **Перевірка JSON-відповіді (як для API):**
`curl -H "Accept: application/json" http://192.168.31.247/notes`

* **Перевірка HTML-відповіді (як для браузера):**
`curl -H "Accept: text/html" http://192.168.31.247/notes`

* **Вивід повного вмісту конкретної нотатки (залежно від її id):**
`curl -X GET http://192.168.31.247/notes/1 -H "Accept: application/json"`

#### 3. Перевірка Health Checks та обмежень Nginx

Назовні Nginx повинен віддавати лише бізнес-логіку. Ендпоінти перевірки стану доступні тільки локально.

* **Локальний виклик (має повернути 200 OK):**
   * `curl http://127.0.0.1:5000/health/alive`,
   * `curl http://127.0.0.1:5000/health/ready`

* **Зовнішній виклик через Nginx (має бути заблоковано - 404 Not Found):**
   * `curl http://192.168.31.247/health/alive`,
   * `curl http://192.168.31.247/health/ready`

#### 4. Перевірка бази даних та міграцій

Перевірка успішного виконання міграційного скрипта та наявності даних.

* **Підключення до бази даних:**

   * `sudo -u postgres psql -d notes_db`

* **Команди всередині psql:**

   * `\dt` — вивести список таблиць (має бути створена таблиця notes).

   * `\d` notes — вивести структуру таблиці notes.

   * `SELECT * FROM notes;` — перевірити наявність створених раніше записів.

   * `\q` — вихід.

#### 5. Перевірка прав доступу (Користувачі та Безпека)

Перевірка обмежень для користувача operator та доступу до журналу.

* **Доступ до файлу gradebook:**

   * `cat /home/student/gradebook` (Успішно для student).

   * `cat /home/student/gradebook` (Помилка Permission denied для teacher, але через sudo можна глянути).

   * `cat /home/student/gradebook` (Помилка Permission denied для operator).

* **Перевірка дозволених команд для operator (виконуються без пароля):**
   
   **Перегляд статусу**

   * `sudo systemctl status mywebapp.service`

   **Зупинка веб-застосунку**

   * `sudo systemctl stop mywebapp.service`

   **Запуск веб-застосунку**

   * `sudo systemctl start mywebapp.service`

   **Перезапуск веб-застосунку**

   * `sudo systemctl restart mywebapp.service`

   **Перезавантаження конфігурації Nginx**
   * `sudo systemctl reload nginx`

* **Перевірка заборонених команд для operator:**

   **Спроба перегляду статусу сокета**
   * `sudo systemctl status mywebapp.socket`
   *(Очікуваний результат: `Sorry, user operator is not allowed to execute '/usr/bin/systemctl status mywebapp.socket' as root...`).*

   **Повний перезапуск Nginx**
   * `sudo systemctl restart nginx`
   *(Очікуваний результат: `Sorry, user operator is not allowed to execute '/usr/bin/systemctl restart nginx' as root...`).*

#### 1. Дослідження Python Application
**Мета:** Проаналізувати вплив оптимізації Dockerfile (кешування шарів) та вибору базового образу (Debian vs Alpine) на розмір та час збірки.
**Директорія експерименту:** [./lab2/python-app-experiment/](./lab2/python-app-experiment/)

**А. Вплив кешування шарів:**

* **Базовий "поганий" образ (python-app-bad):**
  * **Команда збірки:** `docker build -t python-app-bad .`
  * **Розмір:** 1.57 GB | **Час збірки (при зміні коду):** 15.709s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM python:3.13-bookworm
    WORKDIR /app
    COPY . .
    RUN pip install -r requirements/backend.in
    CMD ["python", "spaceship/app.py"]
    ~~~
  * *Аналіз:* Оскільки інструкція `COPY . .` виконувалася перед `RUN pip install`, будь-яка мінімальна зміна у вихідному коді анулювала кеш Docker для наступних кроків. Через це залежності завантажувалися та інсталювалися заново при кожній збірці.

* **Оптимізований образ (python-app-good):**
  * **Команда збірки:** `docker build -t python-app-good .`
  * **Розмір:** 1.57 GB | **Час збірки (при зміні коду):** 0.072s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM python:3.13-bookworm
    WORKDIR /app
    COPY requirements/ requirements/
    RUN pip install -r requirements/backend.in
    COPY . .
    CMD ["python", "spaceship/app.py"]
    ~~~
  * *Аналіз:* Винесення копіювання залежностей в окремий крок перед копіюванням решти коду дозволило ефективно закешувати "важкий" шар із пакетами. Зміна логіки застосунку більше не впливає на цей шар, завдяки чому швидкість повторної збірки зросла у понад 200 разів.

**Б. Debian vs Alpine (з інтеграцією бібліотеки numpy):**

* **Alpine-версія (python-app-alpine-numpy):**
  * **Команда збірки:** `docker build -t python-app-alpine-numpy .`
  * **Розмір:** 296 MB | **Час збірки:** 22.916s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM python:3.13-alpine
    WORKDIR /app
    COPY requirements/ requirements/
    RUN pip install -r requirements/backend.in
    COPY . .
    CMD ["python", "spaceship/app.py"]
    ~~~

* **Debian-версія (python-app-debian-numpy):**
  * **Команда збірки:** `docker build -t python-app-debian-numpy .`
  * **Розмір:** 1.69 GB | **Час збірки:** 25.131s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM python:3.13-bookworm
    WORKDIR /app
    COPY requirements/ requirements/
    RUN pip install -r requirements/backend.in
    COPY . .
    CMD ["python", "spaceship/app.py"]
    ~~~
  * *Аналіз:* Використання Alpine як базового образу дозволило скоротити фінальну вагу контейнера у 5.7 разів (з 1.69 GB до 296 MB). Завдяки наявності сучасних pre-compiled коліс (wheels) під архітектуру musllinux для бібліотеки numpy, Docker не виконував довгу компіляцію з вихідного коду, тому час збірки Alpine-образу залишився швидким і навіть трохи меншим за Debian-образ.

#### 2. Дослідження DNS: Musl (Alpine) vs glibc (Debian)
**Мета:** Перевірити резолвінг кастомних DNS-записів у внутрішній Docker-мережі dns-lab.

* **Команди підготовки середовища:**
  ~~~bash
  docker network create dns-lab
  docker run --rm -it --name dns-server --network dns-lab alpine sh -c "apk add dnsmasq && echo 'address=/myservice.internal.corp/10.0.0.50' > /etc/dnsmasq.conf && dnsmasq -k --log-queries --log-facility=-"
  ~~~
* **Результат:** В обох тестових контейнерах (Ubuntu на базі glibc та Alpine на базі musl) команда `getent hosts myservice.internal` успішно відпрацювала і повернула IP-адресу `10.0.0.50`. Це підтверджує коректність роботи DNS-резолвера у кастомних внутрішніх Docker-мережах незалежно від системної бібліотеки (musl чи glibc).

#### 3. Дослідження Golang Application та Multi-stage builds
**Мета:** Зменшити розмір образу скомпільованого застосунку (проєкт fizzbuzz) за допомогою механізму багатоетапної збірки (Multi-stage).
**Директорія експерименту:** [./lab2/golang-app-experiment/](./lab2/golang-app-experiment/)

* **Базовий образ (golang-app-basic):**
  * **Команда збірки:** `docker build -t golang-app-basic .`
  * **Розмір:** 1.34 GB | **Час збірки:** 1m 4.770s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM golang:1.22
    WORKDIR /app
    COPY go.mod go.sum ./
    RUN go mod download
    COPY . .
    RUN go build -o /fizzbuzz
    CMD [ "/fizzbuzz", "serve" ]
    ~~~
  * *Аналіз:* Фінальний образ містить повне середовище розробки Go (компілятор, SDK, вихідний код, системні утиліти Linux), які абсолютно не потрібні для виконання вже готового бінарника.

* **Multi-stage з використанням scratch (golang-app-scratch):**
  * **Команда збірки:** `docker build -t golang-app-scratch .`
  * **Розмір:** 16.7 MB | **Час збірки:** 13.954s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM golang:1.22 AS builder
    WORKDIR /app
    COPY go.mod go.sum ./
    RUN go mod download
    COPY . .
    RUN CGO_ENABLED=0 GOOS=linux go build -o /fizzbuzz
    
    FROM scratch
    COPY --from=builder /fizzbuzz /fizzbuzz
    COPY --from=builder /app/templates /templates
    CMD [ "/fizzbuzz", "serve" ]
    ~~~
  * *Аналіз:* Розмір образу впав з гігабайтів до лічених мегабайтів, оскільки scratch — це абсолютно порожній контейнер. Було виправлено помилку відсутності статичних файлів шляхом явного копіювання папки `/templates`. Мінус підходу — відсутність будь-якої оболонки (`shell`), що унеможливлює дебаг контейнера зсередини.

* **Multi-stage з використанням distroless (golang-app-distroless):**
  * **Команда збірки:** `docker build -t golang-app-distroless .`
  * **Розмір:** 22.9 MB | **Час збірки:** 24.618s
  * **Конфігурація Dockerfile:**
    ~~~dockerfile
    FROM golang:1.22 AS builder
    WORKDIR /app
    COPY go.mod go.sum ./
    RUN go mod download
    COPY . .
    RUN CGO_ENABLED=0 GOOS=linux go build -o /fizzbuzz
    
    FROM gcr.io/distroless/static-debian12
    COPY --from=builder /fizzbuzz /fizzbuzz
    COPY --from=builder /app/templates /templates
    CMD [ "/fizzbuzz", "serve" ]
    ~~~
  * *Аналіз:* Оптимальний вибір для реальних проєктів. Образ важить усього на 6 МБ більше за scratch, проте містить важливу базову конфігурацію системи (CA-сертифікати для безпечних HTTPS-запитів, налаштування таймзон). При цьому він залишається захищеним від зловмисників через повну відсутність командного інтерпретатора типу `sh`/`bash`.

### Практична частина

**Мета:** Контейнеризувати та забезпечити спільний автоматичний запуск сервісів із Лабораторної роботи №1 за допомогою утиліти Docker Compose.
**Директорія проєкту:** [./lab1/](./lab1/)

**Компоненти інфраструктури розгортання:**
1. **Dockerfile застосунку:** Написано оптимізований за шарами `Dockerfile` на базі стабільного та полегшеного образу `python:3.10-slim`.
2. **Конфігурація оркестрації:** Створено маніфест `docker-compose.yml`, який об'єднує:
   * Базу даних `db` (`postgres:15-alpine`) із збереженням інформації на жорсткому диску хоста за допомогою іменованого сховища `volumes (pgdata)`.
   * Бекенд `backend` (Python), що автоматично проводить міграції схем даних (`migration_script_db.py`) перед запуском веб-сервера.
   * Веб-сервер `nginx` (`nginx:alpine`), налаштований як reverse proxy.
3. **Гнучке керування налаштуваннями:** Конфігураційні файли (`config.json`, `nginx.conf`) монтуються в контейнери "на льоту" у режимі read-only (`:ro`) за допомогою механізму volumes, що дозволило зберегти оригінальну структуру шляхів без переписування коду.

**Особливості та труднощі:**
* **Конфлікт портів:** Під час першого виклику оркестратора виникла помилка `failed to bind host port 0.0.0.0:5432: address already in use`.
* **Причина та рішення:** Порт був зайнятий локальним сервером PostgreSQL, налаштованим під час виконання першої лабораторної роботи. Проблему вирішено зупинкою фонового процесу на хост-машині за допомогою команди: `sudo systemctl stop postgresql`.

**Інструкція з відтворення (запуск проєкту):**
Для запуску всього комплексу сервісів перейдіть у папку `lab1` та виконайте команди:
~~~bash
cd lab1
docker-compose up -d
~~~
Після успішної збірки та ініціалізації (`docker ps`) застосунок стає доступним за адресою: `http://localhost`.

### Розгортання інфраструктури на віртуальній машині (VM)

Цей розділ описує процес перенесення та запуску контейнеризованого комплексу на чистій ізольованій віртуальній машині (наприклад, Ubuntu Server або Linux Mint).

**Крок 1: Налаштування мережі гіпервізора**
Для прямого доступу до веб-сервера з хост-машини, у налаштуваннях мережевого адаптера ВМ (VirtualBox/VMware) замість NAT встановлюється тип підключення: **Мережевий міст (Bridged Adapter)**. Це надає ВМ власну IP-адресу в локальній мережі. *Альтернатива для NAT: налаштовується Port Forwarding (Порт хоста 80 -> Порт гостя 80).*

**Крок 2: Інсталяція середовища у ВМ**
У терміналі віртуальної машини виконується підготовка системних утиліт:
~~~bash
sudo apt update
sudo apt install -y docker.io docker-compose
sudo usermod -aG docker $USER
newgrp docker
~~~

**Крок 3: Клонування та запуск проєкту**
Перенесення робочої папки проєкту у ВМ та ініціалізація інфраструктури:
~~~bash
git clone https://github.com/SirkoKropyvnytskyi/DevOps_labs_2nd_year
cd DevOps_labs/lab1
docker-compose up -d
~~~

**Крок 4: Маршрутизація запитів**
Зсередини ВМ перевірка здійснюється за адресою `http://localhost`. Для доступу з хостового ПК використовується команда `ip a` для визначення IP-адреси ВМ. Після цього веб-інтерфейс стає доступним на хості за адресою `http://<IP_АДРЕСА_ВМ>`.

### Висновки та рекомендації

1. **Оптимізація кешування шарів (Dockerfile):** Порядок виконання інструкцій має вирішальне значення для CI/CD процесів. Винесення відносно статичних компонентів (файли маніфестів залежностей `requirements.txt`, `go.mod`) перед копіюванням динамічного вихідного коду (`COPY . .`) дозволяє зафіксувати шар пакетів у кеші Docker. Це скорочує час повторного білду при зміні бізнес-логіки з 15.709s до 0.072s (у понад 200 разів).

2. **Критерії вибору базових образів:** Базові Linux-дистрибутиви типу Alpine суттєво мінімізують об'єм образу (зменшення у 5.7 разів, з 1.69 GB до 296 MB для Python-інфраструктури). Проте використання Alpine раціональне лише за наявності скомпільованих бінарних пакетів (wheels) під `musl libc`. Для важких математичних або системних бібліотек компіляція з вихідного коду всередині Alpine може критично сповільнити час первинної збірки, тому для них пріоритетними є slim-версії на базі Debian (glibc).

3. **Ефективність багатоетапних збірок (Multi-stage builds):** Для компільованих мов програмування (Golang) архітектурно необхідно розділяти середовище збірки (компілятор, SDK, вихідний код) і середовище виконання готового статичного бінарного файлу. Впровадження інструкцій `AS builder` дозволяє знизити вагу продакшен-образу у 80 разів (з 1.34 GB до 16.7 MB у `scratch`).

4. **Безпека та архітектура продакшен-середовищ:** Використання образів distroless (наприклад, `static-debian12`) є інженерним стандартом безпеки, що переважає мінімалістичний `scratch`. Образи distroless важать усього на ~6 MB більше, але містять критично важливі для корпоративних мереж системні файли (надійні CA-сертифікати для HTTPS, описи локалей, таймзони, не-root користувачів). Повна відсутність командних інтерпретаторів (`sh`/`bash`) у таких контейнерах зводить до нуля можливість виконання довільного коду зловмисниками зсередини контейнера.

5. **Переваги декларативної оркестрації:** Інструмент Docker Compose забезпечує повну відтворюваність та ізоляцію багатоконтейнерних систем хост-рівня. Шляхом автоматичного створення ізольованих віртуальних мереж (bridge), підключення персистентних сховищ даних (volumes) та вбудованого DNS-резолвінгу (зв'язок бекенду з БД за назвою сервісу `db` замість хардкоду IP-адрес) досягається повна незалежність локального розгортання застосунку від конфігурації Host OS.
